# 网络拓扑问题：原因、解决方案与启示

> 日期：2026.6.9
> 阶段：CI/CD 平台基础设施搭建

---

## 问题现象

| 测试项 | 结果 |
|--------|------|
| 笔记本 ← → 鲁班猫 | ✅ USB 共享，可 SSH |
| 主机 WiFi ← → 外网 | ✅ `apt install` 正常 |
| 主机有线 ← → 公司测试环境 | ✅ `172.30.x.x` 可 ping |
| **笔记本 WiFi ← → 主机 WiFi** | **❌ 互 ping 不通** |
| 笔记本 → 各自网关 | ✅ 可 ping |
| 主机 → 各自网关 | ✅ 可 ping |

---

## 原因分析

```
192.168.118.x 子网              192.168.119.x 子网
    笔记本(WiFi)          ✗          主机(WiFi)
         │                               │
      网关 OK                         网关 OK
```

**根因**：公司网络部署了 **WiFi 客户端隔离（Client Isolation / VLAN 隔离）**。不同子网的 WiFi 客户端之间不允许直接通信，各自只能与自己的网关交互。这是企业网络的标准安全策略，并非故障。

**为什么网关能 ping 通但跨子网不通**：网关设在各自子网内，路由不做跨子网转发。笔记本（118.x）和主机（119.x）落到不同 VLAN 段，二层不互通，三层（IP 层）自然无法通信。

---

## 解决方案

### 物理层

**USB 转 RJ45 千兆网口适配器** + 网线直连。

- 成本：20-30 元人民币
- 品牌：绿联（UGREEN）、山泽（SAMZHE）均可
- 协议：USB 3.0 转千兆以太网，Linux 免驱

### 网络层

建立一个只在此链路内使用的私有子网：

| 设备 | 接口 | IP | 子网掩码 | 网关 |
|------|------|----|-----------|------|
| 主机 | USB 转 RJ45 | `10.0.0.1` | `255.255.255.0` | 留空 |
| 笔记本 | 有线 / USB 转 RJ45 | `10.0.0.2` | `255.255.255.0` | 留空 |

**关键操作**：

**主机（Ubuntu）：**
```bash
# 找到新增网卡名称
ip link show

# 配静态 IP 并启用
sudo ip addr add 10.0.0.1/24 dev eth1
sudo ip link set eth1 up
```

**笔记本（Windows）：**
- 控制面板 → 网络连接 → 以太网 → IPv4 属性
- IP: `10.0.0.2`，子网: `255.255.255.0`，网关和 DNS 留空

> ⚠️ 注意：`ip addr add` 临时配置，主机重启后丢失。生产使用需写入 `/etc/netplan/` 固化。

### 最终拓扑

```
鲁班猫 RK3588                   笔记本(Win)                      主机(Ubuntu 22.04)
    ARM64                   10.0.0.2 (有线) ──网线── 10.0.0.1 (USB网口)
  USB共享 ↑              192.168.118.x (WiFi)           192.168.119.x (WiFi)
    ↑   ↓                     外网 OK                      172.30.192.251 (有线)
    └─ SSH ────────────────────→┘                          公司内网
    ←─ 部署 ────────────────────→┘
```

**数据流向**：
1. 笔记本通过私有子网 push 代码到主机的 Gitea
2. Jenkins 在主机上拉取代码、交叉编译
3. 构建产物通过 SCP（笔记本做中转/跳板）部署到鲁班猫
4. 测试结果回传

## 启示与教训

### 1. "先验证网络，再设计架构"

不应该在 ping 都没做的情况下就决定服务部署位置。CI/CD 的本质是**让数据在节点间流动**——代码从开发机到仓库，仓库到构建机，构建产物到目标板。如果其中任意一段链路不通，整个流水线就断了。

**以后每次做架构决策前**：
```
是否所有相关节点都能两两 ping 通？→ 不能 → 先解决连通性 → 再谈架构
```

### 2. "企业网络比你想象的更受限"

公司网络的默认假设不是"放开"，而是"收紧"。WiFi 客户端隔离、外网白名单、端口封锁——这些不是特殊配置，是标准做法。这也正是为什么实习场景（卫星 OS）下同样会遇到这类问题：**受控网络环境本身就是嵌入式系统 CI/CD 需要面对的约束，不是需要绕过的障碍。**

### 3. "多网卡不是过度设计，是必要基础设施"

当初觉得"一台机器配三个网卡"是不是太复杂了。事实证明每块网卡都有独立的职责：

| 网卡 | 职责 | 丢了会怎样 |
|------|------|------------|
| WiFi | 外网访问（apt、docker pull） | 无法装软件、拉镜像 |
| 有线（公司） | 访问公司内网资源 | 与测试环境断联 |
| USB 网口（私有） | 与笔记本的代码通道 | CI/CD 链路断裂 |

三块网卡各自服务一个独立的"安全域"——这是受控网络下的标准架构模式。迁移到卫星 OS 场景后，这个三域模式几乎一定复用。

### 4. "私有子网的 10.0.0.0/8 段是安全选择"

选 `10.0.0.0/24` 而不是 `192.168.x.0/24` 是因为：公司网络已经用了 `192.168.118.0` 和 `192.168.119.0` 段，用 `10.x` 段可以完全避免路由表冲突。如果以后某个网络接口恰好也落到同一段，路由会混乱。

### 5. "笔记本是事实上的网络枢纽，不是边缘角色"

一开始以为笔记本只是开发机，主机才是主角。实际拓扑下，笔记本是**唯一能同时触达外网、主机、鲁班猫的节点**。这意味着：
- 笔记本的稳定性和可用性直接影响 CI/CD
- 代码入库（push 到 Gitea）依赖笔记本
- 部署产物到鲁班猫依赖笔记本
- 必须考虑笔记本不在时的 fallback 方案（长期目标）
